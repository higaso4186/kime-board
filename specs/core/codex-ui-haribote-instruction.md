# Codex 依頼: キメボード UI ハリボテの作成

## 依頼の目的

キメボードの **UI ハリボテ（見た目だけのモック）** を作成してください。  
API や Agent との接続は行わず、**静的レイアウト・ルーティング・モックデータ** で全画面を実装してください。

---

## 前提・参照すべき仕様

- **製品名**: キメボード（会議メモから決裁を抽出し、不足を質問で埋めて「決められる状態」にする B2B SaaS）
- **必読仕様書**:
  - `specs/core/web.yaml` … 画面一覧（S1〜S9）、ナビゲーションシェル、デザイン言語、レイアウト・コンポーネント指定
  - `specs/core/mvp.yaml` … ドメインモデル（Project / Meeting / Decision / Action / Thread / Message）、状態値、用語
- **実装状況**: `spec-impl-map.yaml` によると **apps/web は現時点で仕様のみ**。今回が初実装です。API は別リポジトリまたは `apps/api` に存在する想定ですが、**今回の依頼では API 呼び出しは行わない**でください。

---

## 技術スタック（web.yaml に準拠）

- **フレームワーク**: Next.js（App Router 推奨）
- **UI**: shadcn/ui + Tailwind CSS
- **デザイン**: Modern B2B SaaS / Bento grid / ミニマルなコンテナ
- **参考**: Material Design 3（階層・モーション）
- **フォント**: Inter + Noto Sans JP
- **テーマ**: light、背景 neutral-50、表面 white、アクセント blue-600、角丸 card:12 / input:10 / button:10
- **ビューポート基準**: 1440×900（Desktop 優先。MVP ではモバイル最適化は不要）

## 実装する画面とルート

| ID | 画面名 | ルート |
|----|--------|--------|
| S1 | プロジェクト一覧 | `/projects` |

## ナビゲーションシェル（S2〜S9 で共通）

- **ヘッダー**（高さ 64px）  
  - 左: プロダクトロゴ、プロジェクトスイッチャー  
  - 中央: グローバル検索（placeholder: 「決裁・会議・アクションを検索」）  
  - 右: QuickCreate（新規会議メモ / 新規決裁 / 新規アクション）、通知ベル、ユーザーメニュー  

- **サイドバー**（幅 280px）  
  - プロジェクト: ダッシュボード、決裁バックログ、会議、アジェンダ、管理職ビュー  
  - 設定: ロール/メンバー  

- **右パネル**（幅 360px、デフォルトは閉じる）  
  - タブ: 「チャット」「エージェントログ」  
  - 開くトリガーは「NEEDS_INFO の決裁クリック」「質問を見る」など（ハリボテではボタンで開閉で可）

S1（プロジェクト一覧）はシェルなしの独立レイアウトでよいです。

## 各画面で入れてほしい要素（要約）

- **S1**: トップバー（ロゴ・検索・新規PJボタン）、プロジェクトカードを Bento 風 3 列（PJ名、未決決裁数、READY_TO_DECIDE 数、期限超過数など）。カードクリックで S2 へ遷移。
- **S2**: KPI Bento 4 枚（未決決裁 / 決められる / 期限超過 / 決裁者未設定）、CTA「会議メモを追加」、決裁バックログ表（上位）、「次の会議で決める候補」リスト。
- **S3**: 会議名・日時・参加者・メモ本文のフォーム、「下書き保存」「保存して解析開始」ボタン。右に「入力のコツ」カード。
- **S4**: 会議ヘッダー（会議名・日時）、状態ピル（ANALYZING/DONE/FAILED）、「抽出された決裁」リスト（状態チップ・決裁者・詳細ボタン）、エージェントログカード。
- **S5**: 決裁ヘッダー（タイトル・タグ・状態チップ NEEDS_INFO/READY_TO_DECIDE/DECIDED/REOPEN・決裁者・期限・「確定」ボタン）、選択肢・根拠・前提・反証条件のカード、アクション（Prep/Exec）タブ、「素案生成」ボタン。
- **S6**: チャットコンテナ（エージェントの質問セット＋ユーザー回答の見た目）、右に決裁プレビュー（埋まり具合メーター・READY 条件チェック）。
- **S7**: ビュー切替（カンバン/リスト）、フィルタチップ、カンバン 4 列（NEEDS_INFO / READY_TO_DECIDE / DECIDED / REOPEN）。ドラッグ&ドロップは不要（見た目だけでよい）。
- **S8**: アジェンダ名・対象期間、「自動生成」「保存」ボタン、3 列（決める / 材料を揃える / 再審）、出力プレビュー。
- **S9**: Bento 3 枚（決裁者未設定・期限超過アクション・今週決めるべき決裁）、決裁者未設定テーブル、期限超過アクションリスト、ワンクリック運用ボタン。

いずれも **web.yaml の screens セクション** に詳細があるので、可能な範囲でそれに合わせてください。

## ハリボテの範囲（やること・やらないこと）

### やること

- 上記 9 画面のルーティングとレイアウト
- ナビゲーションシェル（ヘッダー・サイドバー・右パネル）の見た目とリンク
- 各画面のコンポーネント構成（カード、表、リスト、フォーム、チップ、ボタンなど）
- **モックデータ**（日本語でよい）でリスト・表・KPI を表示
- 状態チップ・ラベルの表示（NEEDS_INFO / READY_TO_DECIDE / DECIDED / REOPEN、ANALYZING / DONE / FAILED など）
- 画面間の遷移（リンク・ボタンで S1→S2→S3… などがたどれること）

### やらないこと

- 実 API の呼び出し（Next.js の API Route や Server Actions でバックエンドを叩かない）
- 認証・権限
- ドラッグ&ドロップ（S7 はカンバンの見た目だけ）
- リアルタイム更新・WebSocket
- フォーム送信の実際の保存（送信時は「保存しました」などのトースト or メッセージ表示で代替可）

---

## モックデータの方針

- `projectId` / `meetingId` / `decisionId` は UUID 風の文字列でよい（例: `prj_01`, `mtg_01`, `dcs_01`）。
- 決裁・会議・アクションは 2〜5 件程度のダミーでよい。
- ラベルは `mvp.yaml` や `web.yaml` の用語に合わせる（例: 決裁者未設定、READY_TO_DECIDE、期限超過）。

---

## 成果物の置き場所

- **アプリ**: `apps/web` に Next.js アプリを新規作成
- モックデータは `apps/web` 内の `lib/mock/` や `data/` など、適宜一箇所にまとめてください。
- 既存の `packages/shared` がある場合は、型や定数（ROUTES、DecisionStatus など）を参照してよいです。参照しない場合は、ハリボテ用にローカルで定義しても構いません。
